# Правило распределения Milestone/WBS/Task по спринтам (Этап 5)

Документ описывает формат `schema/sprint_plan.yaml` — эталонного
распределения WBS `e-sklep-b2b` по спринтам — и правила его ручного
заполнения консультантом. Начиная с этой версии Этап 5 — **шаблон, не
алгоритм**: `assemble_plan.py` только читает готовое расписание из
`schema/sprint_plan.yaml`, не вычисляет его (граф зависимостей WBS и
жадная упаковка по трудоёмкости — прежний подход, удалён, см.
`CHANGELOG.md`). Это правило для консультанта (как заполнять/калибровать
`schema/sprint_plan.yaml`), а не изменение структуры шаблона:
`schema/milestones_wbs.yaml` и `tasks/M*_tasks.yaml` этим документом не
меняются и не должны знать о номерах спринтов.

## Источник истины

Comarch не документирует Agile/спринт-планирование (это не описанный
вендором процесс) — правило целиком относится к
`source.type: Internal Project Methodology`, как и остальные
управленческие Task шаблона (M7/M8/M9, см. `docs/source-map.md`).

## Формат `schema/sprint_plan.yaml`

```yaml
sprints:
  - number: 1
    duration_weeks: 2
    wbs: [WBS-1.1, WBS-1.2, WBS-1.3, WBS-1.4, WBS-2.1]
  - number: 2
    duration_weeks: 2
    wbs: [...]
```

- **`number`** — порядковый номер спринта. Спринты обрабатываются
  `assemble_plan.py` в порядке возрастания `number`, независимо от порядка
  записей в файле.
- **`duration_weeks`** — длительность спринта в календарных неделях.
  **Ручное решение консультанта**, не производная от трудоёмкости WBS —
  `assemble_plan.py` не считает часы и не подгоняет длительность под них.
- **`wbs`** — список ID WBS (`schema/milestones_wbs.yaml`), которые целиком
  попадают в этот спринт. **Task наследует спринт своего WBS** — все Task
  одного WBS лежат в одном и том же спринте, WBS между спринтами не
  режется.

### Обязательное покрытие

Каждый WBS из `schema/milestones_wbs.yaml` (все M1–M9) должен встретиться
в `schema/sprint_plan.yaml` **ровно один раз** — не пропущен, не
задублирован в двух спринтах. `assemble_plan.py` проверяет это явно при
каждой сборке плана (и в `--selftest`) и останавливается с явной ошибкой,
а не пропускает WBS молча или использует первое совпадение.

Состав WBS одинаков для всех клиентов независимо от `input-schema.json`
(единственная точка вариативности шаблона, `WBS-6.4`, варьирует состав
Task *внутри* WBS по `integrations`, а не сам факт существования
`WBS-6.4`, см. `docs/selection-rules.md`) — поэтому `schema/sprint_plan.yaml`
общий для всех клиентов, как и `schema/milestones_wbs.yaml`, а не
генерируется отдельно на каждый прогон.

## Даты спринтов

`assemble_plan.py` вычисляет даты последовательно накопительно от
`charter.start_date` конкретного клиента — не по формуле с фиксированным
шагом. Первый спринт (`number: 1`) стартует в `start_date`; дата начала
каждого следующего спринта — дата конца предыдущего + 1 день (спринты идут
подряд, без зазоров). Дата конца спринта = дата начала + `duration_weeks`
недель − 1 день.

Формат имени: `Спринт {N} ({дата начала}–{дата конца})`, без дополнительных
меток — например `Спринт 3 (2026-08-31–2026-09-13)`.

## Завершение проекта (WBS-9.4) и гиперподдержка вне плана

> **Определение (зафиксировано на Этапе 5):** `target_launch_date` = дата
> запуска магазина в эксплуатацию (WBS-9.2, `T-9.2.2` "Объявить магазин
> доступным для покупателей"), а не дата закрытия всего проекта —
> `WBS-9.4` закономерно продолжается после `target_launch_date`.

`WBS-9.4` (Завершение проекта) — обычная запись в `schema/sprint_plan.yaml`,
как любой другой WBS: один спринт, WBS не режется; не входит в проверку
"укладывается ли план в срок" ниже — ожидаемо продолжается после
`target_launch_date`, это не дефицит времени.

Гиперподдержка (согласование периода, мониторинг журнала синхронизации,
завершение периода) в план (`schema/milestones_wbs.yaml`/
`schema/sprint_plan.yaml`) не входит вовсе — WBS-9.3 с этой версии удалён
из шаблона (см. `CHANGELOG.md`). Она ведётся отдельным Epic ("Поддержка"),
который `agent/jira_export.py` создаёт **всегда** при экспорте, без флага
и независимо от конкретного плана — с одной Task-журналом внутри, без
Subtask/Sprint/трудозатрат. Это осознанное архитектурное решение: период
гиперподдержки не привязан к календарным спринтам и трудоёмкости
внедрения, поэтому не пытается притворяться WBS с фиксированной
длительностью.

## Проверка срока (R-7)

После того как каждый WBS получил спринт (и, тем самым, каждый Task —
через наследование), `assemble_plan.py` считает:

- `weeks_used_until_launch` — сумма `duration_weeks` всех спринтов вплоть
  до спринта, в который попал `T-9.2.2` (запуск в эксплуатацию),
  включительно.
- `available_weeks` = `(target_launch_date − start_date) / 7` дней —
  доступные календарные недели конкретного клиента.

Если `weeks_used_until_launch > available_weeks` — генерация **не
блокируется**: `assemble_plan.py` формирует явное предупреждение
(`sprint_plan.warning`) с фактическими числами, и срабатывает риск `R-7`
(`agent/risk-register.yaml`). Решение (сдвинуть `target_launch_date`,
пересмотреть состав `schema/sprint_plan.yaml`, разбить проект на несколько
итераций) — за консультантом/PM, не за агентом.

Ориентиры `sprint_duration_weeks`/`team_capacity_per_sprint`
(`agent/estimation-config.yaml`) больше не читаются `assemble_plan.py` для
расчёта длительности — они остаются справочными числами для консультанта
при ручном заполнении `duration_weeks` в `schema/sprint_plan.yaml`
("сколько недель обычно занимает спринт на N часов при такой команде"), не
жёсткой границей, которую код бы проверял. `remediation_buffer_hours`
(`agent/effort-estimates.yaml`/`assemble_plan.py`, Этап 3) от этого
изменения не зависит — оценка Task типа `remediation` считается так же,
как раньше.

## Изменение шаблона

`schema/sprint_plan.yaml` — **утверждённый шаблон, одинаковый для всех
клиентов** `e-sklep-b2b`, а не отправная точка, которую консультант
перекраивает на каждом новом внедрении. Распределение WBS по спринтам и
`duration_weeks` фиксированы и меняются только явным решением (владелец
шаблона/консультант сознательно пересматривает группировку или сроки для
всех будущих проектов), а не автоматически и не per-клиентски на каждый
прогон `assemble_plan.py`.

Если для конкретного клиента расчётные даты не укладываются в
`target_launch_date` (`sprint_plan.warning`/`R-7`, см. ниже) — это сигнал
консультанту/PM обсудить с заказчиком сдвиг `target_launch_date`, а не
повод перекраивать общий шаблон под один проект. Пересмотр самого
`schema/sprint_plan.yaml` — редкое, осознанное решение, а не штатный шаг
на каждый проект.

Это ручной, а не автоматический процесс — намеренно, после того как
версия с автоматической упаковкой WBS по трудоёмкости (алгоритм) оказалась
избыточной сложностью для целей шаблона (см. `CHANGELOG.md`).

## Границы

- Правило не меняет `schema/milestones_wbs.yaml` и `tasks/M*_tasks.yaml`.
- `schema/sprint_plan.yaml` — часть шаблона (общий для всех клиентов,
  живёт в `schema/`), в отличие от собранного `sprint_plan` внутри
  JSON-плана конкретного клиента (даты и `task_sprint` — per-клиент, из
  `charter.start_date`).
- Правило не переопределяет классификацию Task по типу и не оценивает
  трудозатраты — использует `effort-estimates.yaml` (Этап 3) как есть
  (только для `remediation_buffer_hours` и `sprint_plan.sprints[].hours_used`,
  информационное поле, не влияющее на распределение).
- Автоматический расчёт `duration_weeks`/распределения WBS по трудоёмкости
  (прежний алгоритм) — не вводится обратно этой правкой; если понадобится
  снова, это отдельное предложение для будущей версии, а не точечная правка.
