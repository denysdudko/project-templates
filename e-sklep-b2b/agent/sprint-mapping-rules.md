# Правило распределения Milestone/WBS/Task по спринтам (Этап 5)

Документ описывает, как агент распределяет Task шаблона `e-sklep-b2b`
по спринтам конкретного проекта. Это правило для агента (логика
вычисления), а не изменение структуры шаблона: `schema/milestones_wbs.yaml`
и `tasks/M*_tasks.yaml` этим правилом не меняются и не должны знать о
номерах спринтов.

## Источник истины

Comarch не документирует Agile/спринт-планирование (это не описанный
вендором процесс) — правило целиком относится к
`source.type: Internal Project Methodology`, как и остальные
управленческие Task шаблона (M7/M8/M9, см. `docs/source-map.md`).
Официальная документация Comarch используется только там, где спринт
затрагивает продуктовые ограничения (например, длительность
milestone) — таких пересечений в этом правиле нет.

## Входные данные

| Источник | Что берётся |
|---|---|
| `agent/input-schema.json` | `start_date`, `target_launch_date` — границы проекта |
| `schema/milestones_wbs.yaml` + `tasks/M*_tasks.yaml` | Граф Task: `id`, `depends_on` |
| `agent/effort-estimates.yaml` (Этап 3) | Трудоёмкость каждой Task по типу — не оценивается заново этим правилом |

### Контракт с `effort-estimates.yaml`

Файл Этапа 3 — не таблица `task_id → effort`, а справочник **типов
задач**: 10 `task_types`, каждый с `keywords`, `base_estimate_hours`
(диапазон `{min, max}` в часах) и `task_type_matching_rule`
(эвристика — глагол в `Task.name`, затем чек-листы, затем WBS-контекст).
Правило распределения по спринтам использует его так:

1. **Классифицировать** каждую Task по `task_type_matching_rule` из
   `effort-estimates.yaml` (само правило классификации не переопределяется
   здесь — оно уже описано в файле Этапа 3).
2. **Взять точечную оценку** трудоёмкости Task как
   `(base_estimate_hours.min + base_estimate_hours.max) / 2` часов —
   для целей планирования спринтов используется середина диапазона
   (константа v1: правило "средняя точка диапазона", а не min/max).
3. **Особые типы** (не укладываются в обычную точечную оценку):
   - **`remediation`** (`base_estimate_hours: null`) — по самому файлу
     Этапа 3 оценивается резервом, не point-оценкой. Правило: Task этого
     типа получает не оценку из справочника, а фиксированный
     `remediation_buffer_hours` (константа v1, см. ниже) и участвует в
     обычном жадном распределении (шаг 3 алгоритма) с этой величиной.
   - **`support_monitoring`** (T-9.3.2) — оценка в файле Этапа 3 дана как
     "часов в неделю на период гиперподдержки", а не на Task целиком, и
     сам период (T-9.3.1) согласовывается с заказчиком по ходу проекта,
     а не задан полем во входной схеме. Это правило не пытается
     bin-packing'ом раскидать T-9.3.2 по нескольким будущим спринтам —
     см. отдельный пункт "Гиперподдержка после запуска" ниже.
   - Task без записи среди `task_types` (эвристика классификации не
     дала совпадения) не может быть распределена — это ошибка входных
     данных/классификации, а не повод пропустить Task молча.

## Константы v1 (не входят в `input-schema.json`)

`sprint_length_weeks` и `capacity_per_sprint_hours` вынесены в
`agent/estimation-config.yaml` (`sprint_duration_weeks`/
`team_capacity_per_sprint`) — раньше были захардкожены в `assemble_plan.py`,
теперь конфигурируются без изменения кода. Полей под них в
`input-schema.json` по-прежнему нет — `estimation-config.yaml` задаёт
значения по умолчанию для шаблона в целом (не per-клиент), а не заменяет
`input-schema.json`; добавлять новое поле в `input-schema.json` без явного
согласования нельзя (см. `CLAUDE.md`).

- **`sprint_duration_weeks = 2`** (`estimation-config.yaml`) — длительность
  спринта. Кандидат в input-schema v2 (например,
  `methodology.sprint_length_weeks`), если потребуется конфигурировать
  per-проект, а не только per-шаблон.
- **`team_capacity_per_sprint = 80`** часов (`estimation-config.yaml`) —
  доступная трудоёмкость на спринт, из допущения "1 консультант, полная
  занятость, 5 рабочих дней в неделю, 8 часов в день". Кандидат в
  input-schema v2 (например, `team_size` / `consultants_count`), если
  потребуется учитывать команду из нескольких консультантов или неполную
  занятость.
- **`remediation_buffer_hours = 0.2 * capacity_per_sprint_hours`**
  (по умолчанию `16` часов = 2 рабочих дня, всё ещё константа в коде, не в
  `estimation-config.yaml`) — резерв на Task типа `remediation`
  (устранение замечаний, T-7.3.1), т.к. справочник Этапа 3 сознательно не
  даёт для этого типа `base_estimate_hours`. Доля `0.2` — эвристика этого
  правила (v1), не факт из документации или из `effort-estimates.yaml`.

Расширение на per-клиентскую конфигурацию (через `input-schema.json`) —
предложение для v2, а не вводится этой правкой.

## Алгоритм распределения (v2 — упаковка WBS целиком)

**Изменение архитектуры алгоритма**, не точечный фикс: единица упаковки —
**WBS целиком** (сумма трудоёмкости всех его Task), а не отдельная Task, и
длительность спринта **переменная**, а не фиксированный грид по
`sprint_duration_weeks`. Причина: жадное распределение отдельных Task
могло раскидать Task одного WBS по нескольким спринтам, что не отражает
реальную практику (WBS сдаётся консультантом как единый результат).
`sprint_duration_weeks`/`team_capacity_per_sprint` остаются целевыми
ориентирами упаковки (см. "Константы v1" выше), но перестают быть жёсткой
границей спринта.

1. **Построить WBS-уровневый граф зависимостей**: WBS `A` зависит от WBS
   `B`, если хотя бы одна Task из `A` имеет `depends_on` на Task из `B`
   (`B ≠ A` — зависимости внутри одного WBS не считаются межблочными).
   `used_by` по-прежнему не используется как источник порядка (см. выше).
   WBS-9.3/WBS-9.4 (Гиперподдержка/Завершение проекта) не входят в этот
   граф — см. "Гиперподдержка после запуска" ниже, они по-прежнему
   планируются по отдельной Task, как в v1.

2. **Определить порядок WBS** топологической сортировкой этого графа.
   Тай-брейк для несвязанных WBS — порядок их следования в шаблоне
   (Milestone → WBS, как в `schema/milestones_wbs.yaml`) — тот же принцип,
   что раньше применялся к отдельным Task.

3. **Жадно набирать WBS-блоки в текущий спринт** в этом порядке:
   - Очередной WBS целиком добавляется в текущий открытый спринт, если
     его суммарная трудоёмкость помещается в остаток
     `capacity_per_sprint_hours`.
   - Как только очередной WBS **целиком** не помещается в остаток —
     текущий спринт закрывается (даже если его фактическая длительность
     меньше номинальных `sprint_duration_weeks`) и открывается новый.
   - Если WBS сам по себе больше `capacity_per_sprint_hours` — он **не
     режется**: получает спринт целиком под себя, а спринт растягивается
     под его объём (см. шаг 4 про длительность).
   - Топологический порядок WBS (шаг 1–2) в сочетании с этой жадной,
     строго последовательной упаковкой сам по себе гарантирует, что
     зависимость WBS уже размещена в спринте `≤` текущего — отдельный
     `min_sprint`-floor (как для отдельных Task в v1) здесь не нужен.

4. **Длительность каждого спринта** = `ceil(часы_в_спринте /
   недельная_нагрузка)` недель, где `недельная_нагрузка =
   team_capacity_per_sprint / sprint_duration_weeks` (по умолчанию
   `80 / 2 = 40` ч/неделю) — не фиксированные `sprint_duration_weeks`.
   Спринт, зарезервированный только под `T-9.3.2` (без обычных Task,
   `hours_used = 0` — см. "Гиперподдержка после запуска" ниже), получает
   номинальную `sprint_duration_weeks`: формула для нуля часов календарно
   не определена, а длительность периода гиперподдержки как таковая этим
   правилом не выводится из трудозатрат. Даты спринтов вычисляются
   последовательно накопительно: следующий спринт стартует сразу после
   конца предыдущего (не по формуле `start_date + (N-1) × const`, как в v1).

5. **Посчитать доступные недели проекта:**
   `available_weeks = (target_launch_date - start_date) / 7 дней`.

6. **Сравнить и зафиксировать результат** — для WBS вплоть до WBS-9.2
   включительно (запуск в эксплуатацию; см. "Гиперподдержка после
   запуска" ниже про WBS-9.3/9.4). Количество спринтов само по себе
   **больше не показатель** при переменной длине — сравниваются
   фактические недели:
   - `weeks_used_until_launch ≤ available_weeks` → план распределения
     фиксируется как есть (даты/номера спринтов — результат генерации
     для конкретного проекта, в шаблон не переносятся).
   - `weeks_used_until_launch > available_weeks` → генерация **не
     блокируется**; агент формирует явное предупреждение (risk/flag) с
     фактическими числами (`weeks_used_until_launch`, `available_weeks`,
     дефицит в неделях), чтобы консультант/PM принял решение (сдвинуть
     `target_launch_date`, увеличить `team_capacity_per_sprint`,
     пересмотреть объём). Решение — за человеком, не за агентом.

## Гиперподдержка после запуска (WBS-9.3, WBS-9.4)

> **Определение (зафиксировано на Этапе 5):** `target_launch_date` = дата
> запуска магазина в эксплуатацию (WBS-9.2), а не дата закрытия всего
> проекта — "укладывается в срок" на шаге 5 проверяется только до
> запуска; WBS-9.3/WBS-9.4 закономерно продолжаются после
> `target_launch_date`, и Этап 6 (сборка Charter) должен явно показать
> заказчику эту разницу, а не подразумевать её.

`target_launch_date` соответствует запуску магазина в эксплуатацию
(WBS-9.2, T-9.2.2 "Объявить магазин доступным для покупателей"), а не
концу проекта — WBS-9.3 (Гиперподдержка) и WBS-9.4 (Завершение проекта)
по смыслу шаблона идут уже после этой даты. Правило учитывает это:

- **WBS-9.3/WBS-9.4 не участвуют в WBS-упаковке** (шаги 1–4 выше) —
  планируются **по отдельной Task**, тем же жадным алгоритмом, что и в
  v1 (Task может быть назначена в спринт `N` только если её
  `depends_on`-предшественники уже размещены в спринте `≤ N`), начиная
  сразу после спринта, в который попал T-9.2.2. Причина: T-9.3.1 не
  имеет `depends_on` в шаблоне (иначе встал бы в спринт 1 вместе с
  основной упаковкой), и период гиперподдержки по смыслу — открытое
  продолжение работ, а не единый сдаваемый блок, как остальные WBS.
- Они **не входят** в сравнение `weeks_used_until_launch` с
  `available_weeks` на шаге 6 — ожидаемо выходят за `target_launch_date`,
  это не дефицит времени.
- Разовые Task WBS-9.3/9.4 (T-9.3.1, T-9.3.3, T-9.4.1, T-9.4.2)
  распределяются обычным образом (точечная оценка по типу).
- T-9.3.2 (`support_monitoring`, периодическая нагрузка, а не разовая
  Task) не оценивается point-оценкой и не проверяется на "укладывается/не
  укладывается": длительность периода гиперподдержки — предмет
  согласования с заказчиком в T-9.3.1, а не поле `input-schema.json`.
  Для целей плана агент резервирует под T-9.3.2 ровно один спринт сразу
  после спринта T-9.3.1, с трудоёмкостью
  `(support_monitoring.min + support_monitoring.max) / 2 часов в неделю
  × sprint_duration_weeks`, и явно указывает в выводе, что фактическая
  длительность гиперподдержки определяется по итогам T-9.3.1 и может
  продолжаться дольше зарезервированного спринта — без предупреждения о
  нехватке времени, т.к. это ожидаемое продолжение работ после запуска.
  Поскольку в этом зарезервированном спринте нет обычных Task
  (`hours_used = 0`), его календарная длительность — номинальные
  `sprint_duration_weeks` (см. шаг 4 выше), а не производная от часов.

## Именование спринтов

Каждый спринт в `sprint_plan.sprints` получает поле `name` — вычисляется
из уже существующих данных, без нового поля во входной схеме:

- Дата начала спринта `1` = `start_date` проекта; дата начала спринта
  `N > 1` = дата конца спринта `N-1` + 1 день (спринты идут подряд, без
  зазоров — не по формуле `start_date + (N-1) × const`, как в v1).
- Длительность спринта — переменная, `sprint_duration_weeks_for(hours_used)`
  (см. "Алгоритм распределения", шаг 4); дата конца = дата начала +
  длительность − 1 день (последний день спринта, не первый день следующего).
- Формат не изменился: `Спринт {N} ({дата начала}–{дата конца})`, например
  `Спринт 3 (2026-08-31–2026-09-13)`.

Спринт, целиком зарезервированный под `T-9.3.2` (`support_monitoring`,
`reserve_exclusive` — см. "Гиперподдержка после запуска" выше), получает в
скобках дополнительную метку — название WBS-9.3 из
`schema/milestones_wbs.yaml` ("Гиперподдержка" в шаблоне v1):
`Спринт 5 (2026-09-28–2026-10-11, Гиперподдержка)`. Это единственный
случай пометки: спринт под T-9.3.2 по построению не делит трудоёмкость ни
с одной другой Task, поэтому метка однозначно описывает его целиком.

`remediation` (T-7.3.1, устранение замечаний) — намеренно **не** повод для
метки: эта Task участвует в обычном жадном распределении (шаг 3
алгоритма) и делит спринт с другими Task на общих основаниях, а не
резервирует его целиком — подписать такой спринт как "Устранение
замечаний" значило бы приписать ему смысл, которого нет у остальных Task
в этом же спринте.

`generate_client_document.py` (Этап 7.5) показывает клиенту `name`
спринта вместо голого номера в таблице плана; CSV-таблица правок (Этап
7.5/7.6) продолжает использовать голый номер в колонке `Спринт` — она
должна оставаться редактируемой простым числом и сливаться обратно через
`merge-corrections.py`.

## Границы

- Правило не меняет `schema/milestones_wbs.yaml` и `tasks/M*_tasks.yaml`.
- Номера спринтов не фиксируются в шаблоне — вычисляются агентом
  индивидуально для каждого проекта на основе его `input-schema.json`.
- Правило не переопределяет классификацию Task по типу и не оценивает
  трудозатраты заново — использует `effort-estimates.yaml` (Этап 3) как есть.
- Любое расширение (новые поля `input-schema.json` под длительность
  спринта/capacity команды, учёт параллельных консультантов, учёт
  нерабочих дней/праздников, явное поле `task_type` в `tasks/M*_tasks.yaml`)
  не вводится по ходу — выносится как предложение для v2.
