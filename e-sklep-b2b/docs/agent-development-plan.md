# План разработки агента-планировщика (Comarch e-Sklep B2B)

Документ фиксирует последовательность этапов создания агента, который берёт
`agent/input-schema.json` конкретного клиента и шаблон Milestones/WBS/Task
(`schema/milestones_wbs.yaml` + `tasks/M*_tasks.yaml`) и генерирует из них
план внедрения. Ничего из перечисленного ниже не меняет утверждённую структуру
шаблона — `schema/milestones_wbs.yaml` и состав `tasks/M*_tasks.yaml` остаются
источником истины для WBS/Task; агент только читает и адаптирует их под клиента
(см. `docs/principles.md`, иерархия источников истины).

Статусы: **готово** / **не начато**. Этап переводится в «готово», когда для него
создан и согласован соответствующий артефакт в `e-sklep-b2b/`.

## Этап 1 — Фиксация входной схемы агента
**Статус: готово.** Артефакт: `agent/input-schema.json`.

Формализует вход клиента (название проекта, заказчик, ERP и его версия,
количество пользователей, интеграции, особенности, даты) как строгий
JSON-контракт — это данные для агента, а не текст, который LLM интерпретирует
свободно.

## Этап 2 — Логика отбора шаблона (не генерация с нуля)
**Статус: готово.** Артефакт: `docs/selection-rules.md`.

Правило маппинга «входные параметры → состав WBS/Task»: core-объём (M1–M9,
WBS-1.1–9.4) неизменен и присутствует всегда; единственная точка вариативности —
WBS-6.4, управляемая полем `integrations` (пустой список — задача-заглушка
T-6.4.2 остаётся framework-only; непустой — на каждую интеграцию разворачивается
child-экземпляр слота T-6.4.2, а не новая WBS). Остальные поля входа (`features`,
`users_count`, `dates`, `erp.version`) — метаданные для контента, не селекторы
структуры.

## Этап 3 — Модель оценки трудозатрат
**Статус: готово.** Артефакт: `agent/effort-estimates.yaml`.

Справочник «тип задачи → базовая оценка часов», отдельный от самого шаблона
задач (в `tasks/M*_tasks.yaml` оценок часов нет и не появляется). Агент
классифицирует конкретную Task по эвристике (ключевые слова в названии +
наличие/характер чек-листов) и подбирает оценку по типу, а не придумывает её
заново для каждой задачи. Источник оценок — опыт консультантов
(`source.type: Internal Project Methodology`), так как документация Comarch
нормативов трудозатрat не содержит.

## Этап 4 — Реестр рисков
**Статус: готово.** Артефакт: `agent/risk-register.yaml`.

Статичная таблица «риск → условие появления по параметрам входа»: 7 рисков,
выведенных из фактических ограничений и зависимостей в M1–M9 (не придуманных
абстрактно), каждый с проверяемым `condition` на полях `agent/input-schema.json`
(`erp.version`, `integrations`, `features`, `users_count`, `start_date`/
`target_launch_date`) и `related_wbs`. Риск неполноты фотоконтента/каталога не
включён — для него нет проверяемого поля во входной схеме (см.
`v2_proposals` в файле реестра). Вероятность/критичность риска не оценивается —
только факт условия появления.

## Этап 5 — Правило Milestone → Sprint
**Статус: готово.** Артефакт: `agent/sprint-mapping-rules.md`.

Единый граф зависимостей по `depends_on` через все M1–M9 (включая
межмилестоновые связи), топологический порядок с тай-брейком по порядку в
шаблоне, жадное распределение по спринтам с трудоёмкостью из
`agent/effort-estimates.yaml` (классификация Task по типу + средняя точка
диапазона часов; отдельные правила для `remediation` — резерв, и
`support_monitoring` — недельная нагрузка гиперподдержки после
`target_launch_date`, не участвующая в проверке укладывания в срок).
Длительность спринта (2 недели) и capacity (1 консультант × 8ч × 5 дней)
зафиксированы как константы v1 — соответствующих полей в
`input-schema.json` нет, добавление не выполнялось без согласования. Если
расчётный объём не укладывается в `target_launch_date` — не блокирует
генерацию, формирует предупреждение с фактическими числами. Номера
спринтов в шаблон не переносятся.

## Этап 6 — Сборка выходного пакета
**Статус: готово.** Артефакт: `agent/assemble_plan.py` (+ `agent/examples/`).

Исполняемый Python-пайплайн (не только описание правилом, как на Этапах
2/5): Charter → Milestones → WBS → Tasks (из шаблона + вариативность
WBS-6.4 по `docs/selection-rules.md`) → Dependencies (`depends_on` из
шаблона, без изменений) → Оценки трудозатрат (`agent/effort-estimates.yaml`,
Этап 3) → Sprint-план (`agent/sprint-mapping-rules.md`, Этап 5) → Риски
(`agent/risk-register.yaml`, Этап 4, включая R-7 по факту готового
Sprint-плана) → Deliverables (агрегация `verification_checklist`, без
новых формулировок). Sprint-план идёт раньше рисков не случайно: R-7
отбирается по флагу "не укладывается в срок" из Sprint-плана, и без него
этот флаг не существует.

LLM подключается только последним шагом (`adapt_wording_with_llm`) и
только как контракт: разрешено переписывать исключительно текстовые поля
(Charter.description/objective, Task.description,
interview_checklist/verification_checklist); попытка изменить
id/depends_on/used_by/состав Milestone/WBS/Task отклоняется явной
ошибкой (структурный fingerprint до/после сверяется программно), а не
проверяется на словах. Без подключённого LLM-клиента шаг — no-op
passthrough (в репозитории нет живого LLM API).

`--selftest` прогоняет классификацию всех реальных Task шаблона по
`effort-estimates.yaml`, сверяет её с `examples` из самого справочника,
проверяет обе ветки WBS-6.4 (пустой/непустой список интеграций, включая
интеграцию вне `KNOWN_INTEGRATION_DOCS`), ветку "не укладывается в срок"
(R-7) и отказ LLM-контракта при попытке сломать структуру. Пример
прогона на `agent/examples/client-abc.input.json` (тот же условный
клиент "ABC", что фигурировал как пример в `agent/input-schema.json`)
сохранён в `agent/examples/client-abc.plan.json`.

Известное ограничение, найденное при реализации (не исправлялось в этом
проходе — см. `open-issues.md`): у 7 Task keyword-эвристика
`effort-estimates.yaml` не даёт совпадения (глагол не входит в keywords
или не в начале названия) — в `assemble_plan.py` заведён
`MANUAL_TASK_TYPE_OVERRIDES` с обоснованием для каждой, `effort-estimates.yaml`
не менялся.

Этап 8 (Jira-экспорт) в этот пайплайн не входит.

## Этап 7 — Валидатор перед выдачей
**Статус: готово.** Артефакт: `agent/validate_plan.py`.

Линтер над выходом `assemble_plan.py` (Этап 6) — отчёт для консультанта/PM
перед экспортом в Jira (Этап 8), не гейт: ничего не блокирует и не
переписывает, только явно перечисляет находки. Пять проверок: (1)
`integrity` — все ID из `depends_on`/`used_by` существуют среди реальных
Task/WBS ID плана; (2) `source_url` — каждый `source.url` принадлежит
домену/разделу официальной документации Comarch (по домену + форме пути,
т.к. реальные статьи — конкретные страницы, а не сами 4 URL из
`docs/principles.md`); (3) `source` — ни одна Task не без `source`, кроме
`Internal Project Methodology`; (4) `checklist_shape` — `interview_checklist`/
`verification_checklist` строго `list[str]` (автоматизирует находку,
исправленную вручную перед Этапом 7 — незакавыченный `": "` внутри пункта
чек-листа YAML превращает его в dict); (5) `cross_section` — каждая Task
из `milestones` присутствует в `dependencies`, `effort_estimates` и
`sprint_plan.task_sprint`, и наоборот.

`--selftest` намеренно ломает копию `agent/examples/client-abc.plan.json`
под каждую из 5 проверок и подтверждает, что она срабатывает, плюс
regression-прогон на самом `client-abc.plan.json` — ожидается 0 находок.
При реализации найден и исправлен (в `assemble_plan.py`, не в шаблоне)
второй случай той же проблемы, что у `T-6.5.1.depends_on` (Этап 6) —
`T-6.4.1.used_by` тоже ссылался на слот `T-6.4.2` (см. `open-issues.md`
п.4). Отчёт по примеру сохранён в
`agent/examples/client-abc.validation-report.txt`.

Изначально (на момент готовности Этапа 7) регрессия давала 16 находок —
`open-issues.md` п.1 (`used_by` на Milestone ID вместо Task/WBS ID).
Пункт закрыт отдельным проходом: все 16 случаев в `tasks/M*_tasks.yaml`
заменены на конкретного потребителя (Task ID, определённый по
`depends_on` следующей задачи или по текстовой ссылке в `description`,
где явного `depends_on` не было — см. `CHANGELOG.md`), `--selftest`
обновлён под чистую регрессию (0 находок).

## Этап 7.5 — Документ для согласования с заказчиком
**Статус: в работе.** Артефакт: `agent/generate_client_document.py`.

Отдельный шаг между валидацией (Этап 7) и Jira-экспортом (Этап 8) — не
заменяет и не предвосхищает ни тот, ни другой: берёт уже собранный и
провалидированный план и рендерит его в человекочитаемый Markdown-документ
для согласования сроков/объёма с заказчиком. Формат — markdown-таблица
(git-diffable), без генерации docx/xlsx.

Содержание: шапка из `charter` (проект, заказчик, цель, даты, с явным
пояснением, что дата запуска — не конец проекта); таблица Milestone → WBS →
Task с номером спринта (`sprint_plan.task_sprint`) и зависимостями,
показанными названиями задач, а не `T-ID`; критерии завершения этапа —
агрегированные `deliverables` (`verification_checklist`), без
`interview_checklist`; риски — только реально сработавшие для конкретного
клиента (`plan.risks`, не весь `risk-register.yaml`), без `id`/`condition`/
`source`, со скобочными техническими уточнениями (`WBS-ID`, `T-ID`, пути к
файлам), вырезанными как безопасно необязательные; итоговая сводка —
число спринтов до запуска и всего, длительность, `sprint_plan.warning`,
если есть. Именованные интеграции (WBS-6.4, `T-6.4.2.1..N`) в таблице,
критериях завершения и зависимостях схлопываются в один плейсхолдер
(`Настроить интеграцию [Компонент] согласно документации Comarch`) — их
список формально фиксируется на интервью (T-1.3.1/T-1.3.2), а не на момент
согласования плана; правка только рендеринга, в JSON-плане (Этап 6)
`T-6.4.2.1..N` остаются как есть.

Второй артефакт того же скрипта (`--corrections-output`) — CSV-таблица
правок: по одной строке на каждую реальную Task плана (Task ID, WBS ID,
Milestone ID, Название, Спринт, Include, Comment), `Include` = `yes` по
умолчанию, `Comment` пуст. В отличие от markdown-документа, именованные
интеграции в ней НЕ схлопнуты — таблица технической, а не презентационная,
и должна сливаться обратно по настоящим Task ID (Этап 7.6).

Известное ограничение (не устранялось в этом проходе): часть текстов
`risk` в `risk-register.yaml` содержит инлайновый (не в скобках) технический
жаргон (`integrations`, `source.url`, `multiple_warehouses` и т.п.), который
`clean_business_text()` не трогает — вырезание отдельных слов из середины
предложения регулярным выражением ломает грамматику чаще, чем помогает;
полноценная адаптация формулировок под клиента — задача LLM-слоя (Этап 6),
не детерминированного скрипта. `--selftest` проверяет структурную утечку
(`interview_checklist`, сырые ключи типа `"url":`, `id` риска), а не
идеальность прозы. Статус "в работе", а не "готово", пока это ограничение
не рассмотрено отдельно.

`--selftest` прогоняется на `agent/examples/client-abc.plan.json`; пример
документа сохранён в `agent/examples/client-abc.agreement.md`, пример
CSV-таблицы правок — в `agent/examples/client-abc.corrections.csv`.

## Этап 7.6 — Приём правок и слияние с базовым планом
**Статус: готово.** Артефакт: `agent/merge-corrections.py`.

Принимает базовый JSON-план (Этап 6) и отредактированную CSV-таблицу правок
(Этап 7.5) и сливает их по Task ID в один план — именно он, а не исходный
JSON Этапа 6, идёт на вход Этапу 8 (Jira-экспорт). Правила слияния: `Include`
= `no`, ИЛИ строка с Task ID вообще отсутствует в таблице (удалена) → Task
исключается из плана; непустые `Название`/`Спринт` из таблицы применяются к
соответствующей Task; строка без Task ID добавляется как новая custom Task —
обязателен существующий WBS ID (новый WBS/Milestone через таблицу правок не
вводится — отдельное решение, не для этого шага), `source.type: Internal
Project Methodology` и `added_by: client_approval_process`.

При исключении Task её ID точечно вычищается из `dependencies`/
`effort_estimates`/`sprint_plan.task_sprint`/`deliverables` (это производные
от `milestones` разделы), но `depends_on`/`used_by` оставшихся Task,
ссылающиеся на исключённый ID, не переписываются автоматически — разрыв
графа зависимостей, если он возник, обязан явно всплыть в отчёте
`validate_plan.py` (Этап 7, `integrity`), а не потеряться молча. По той же
логике у custom Task намеренно не выставляется `effort_estimates` (в
таблице правок нет данных для оценки трудозатрат) — его отсутствие тоже
видно как находка `cross_section`.

После слияния скрипт обязательно повторно прогоняет `validate_plan.py` на
получившемся плане и печатает и журнал слияния, и отчёт валидатора.

`--selftest` прогоняется на `agent/examples/client-abc.plan.json`: пустая/
неизменённая таблица не меняет план; `Include=no` и отсутствующая строка
одинаково исключают Task; исключение Task с зависимыми даёт находку
`integrity`; правки Название/Спринт применяются; новая custom Task в
существующем WBS добавляется корректно и даёт находку `cross_section` по
`effort_estimates`; строка с несуществующим WBS ID отклоняется без создания
нового WBS. Прогон на примере (client ABC: исключены 2 Task, изменены 2
спринта, добавлена 1 custom Task) сохранён в
`agent/examples/client-abc.corrections.edited.csv`,
`agent/examples/client-abc.merged-plan.json` и
`agent/examples/client-abc.merged-plan.validation-report.txt`.

## Этап 8 — Jira-экспорт
**Статус: не начато.**

Маппинг: Milestone → Epic, Task ID → Issue, WBS → Label/Component,
`interview_checklist`/`verification_checklist` → описание задачи в Jira.
Отдельный шаг, срабатывающий только по явному подтверждению утверждённого
плана человеком — не выполняется автоматически вместе со сборкой пакета
(Этап 6). На вход берётся результат слияния правок (Этап 7.6), а не
исходный JSON Этапа 6.

---
Изменения в этом плане (новые этапы, переформулировка существующих) фиксируются
здесь же и отражаются короткой записью в `CHANGELOG.md`, как и любое другое
изменение шаблона.
