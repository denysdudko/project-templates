# План разработки агента-планировщика (Comarch e-Sklep B2B)

Документ фиксирует последовательность этапов создания агента, который берёт
`agent/input-schema.json` конкретного клиента и шаблон Milestones/WBS/Task
(`schema/milestones_wbs.yaml` + `tasks/M*_tasks.yaml`) и генерирует из них
план внедрения. Ничего из перечисленного ниже не меняет утверждённую структуру
шаблона — `schema/milestones_wbs.yaml` и состав `tasks/M*_tasks.yaml` остаются
источником истины для WBS/Task; агент только читает и адаптирует их под клиента
(см. `docs/principles.md`, иерархия источников истины).

Статусы: **готово** / **не начато**. Этап переводится в «готово», когда для него
создан и согласован соответствующий артефакт в `e-sklep-b2b/`.

## Этап 1 — Фиксация входной схемы агента
**Статус: готово.** Артефакт: `agent/input-schema.json`.

Формализует вход клиента (название проекта, заказчик, ERP и его версия,
количество пользователей, интеграции, особенности, даты) как строгий
JSON-контракт — это данные для агента, а не текст, который LLM интерпретирует
свободно.

## Этап 2 — Логика отбора шаблона (не генерация с нуля)
**Статус: готово.** Артефакт: `docs/selection-rules.md`.

Правило маппинга «входные параметры → состав WBS/Task»: core-объём (M1–M9,
WBS-1.1–9.4) неизменен и присутствует всегда; единственная точка вариативности —
WBS-6.4, управляемая полем `integrations` (пустой список — задача-заглушка
T-6.4.2 остаётся framework-only; непустой — на каждую интеграцию разворачивается
child-экземпляр слота T-6.4.2, а не новая WBS). Остальные поля входа (`features`,
`users_count`, `dates`, `erp.version`) — метаданные для контента, не селекторы
структуры.

## Этап 3 — Модель оценки трудозатрат
**Статус: готово.** Артефакт: `agent/effort-estimates.yaml`.

Справочник «тип задачи → базовая оценка часов», отдельный от самого шаблона
задач (в `tasks/M*_tasks.yaml` оценок часов нет и не появляется). Агент
классифицирует конкретную Task по эвристике (ключевые слова в названии +
наличие/характер чек-листов) и подбирает оценку по типу, а не придумывает её
заново для каждой задачи. Источник оценок — опыт консультантов
(`source.type: Internal Project Methodology`), так как документация Comarch
нормативов трудозатрat не содержит.

## Этап 4 — Реестр рисков
**Статус: готово.** Артефакт: `agent/risk-register.yaml`.

Статичная таблица «риск → условие появления по параметрам входа»: 7 рисков,
выведенных из фактических ограничений и зависимостей в M1–M9 (не придуманных
абстрактно), каждый с проверяемым `condition` на полях `agent/input-schema.json`
(`erp.version`, `integrations`, `features`, `users_count`, `start_date`/
`target_launch_date`) и `related_wbs`. Риск неполноты фотоконтента/каталога не
включён — для него нет проверяемого поля во входной схеме (см.
`v2_proposals` в файле реестра). Вероятность/критичность риска не оценивается —
только факт условия появления.

## Этап 5 — Sprint-план как шаблон (не алгоритм)
**Статус: готово.** Артефакты: `schema/sprint_plan.yaml`,
`agent/sprint-mapping-rules.md`.

**Изменение архитектуры (не точечный фикс):** прежний алгоритм (WBS-уровневый
граф зависимостей + жадная упаковка по трудоёмкости с переменной длиной
спринта) удалён. Вместо вычисления `assemble_plan.py` **читает готовое
расписание** из `schema/sprint_plan.yaml` — эталонного распределения всех
WBS (M1–M9) по спринтам, заполненного вручную консультантом (см.
`agent/sprint-mapping-rules.md` — формат файла и правила калибровки). Task
наследует спринт своего WBS целиком — WBS между спринтами не режется.
`assemble_plan.py` проверяет только, что каждый реальный WBS плана покрыт
ровно одним спринтом (не пропущен, не задублирован) — это единственная
проверка, не вычисление. `WBS-9.4` (Завершение проекта) — обычная запись
шаблона, как любой другой WBS, без отдельной Task-level логики.
Гиперподдержка (бывший `WBS-9.3`) с этой версии в план не входит вовсе —
удалена из `schema/milestones_wbs.yaml`, ведётся отдельным Epic "Поддержка"
в Jira (Этап 8, создаётся всегда, без флага).

Даты каждого спринта конкретного клиента вычисляются последовательно
накопительно от `charter.start_date`: первый спринт стартует в `start_date`,
следующий — сразу после конца предыдущего; длительность — `duration_weeks`
из `schema/sprint_plan.yaml` (ручное решение консультанта, не производная от
трудоёмкости). Имя — `Спринт {N} ({дата начала}–{дата конца})`, без меток.

Проверка срока (`R-7`) не изменилась по смыслу: сумма `duration_weeks`
спринтов вплоть до спринта запуска (`T-9.2.2`, WBS-9.2) сравнивается с
доступными календарными неделями `start_date → target_launch_date`; при
дефиците генерация не блокируется, формируется предупреждение и срабатывает
риск `R-7`. `sprint_duration_weeks`/`team_capacity_per_sprint`
(`agent/estimation-config.yaml`) больше не читаются кодом для расчёта —
остаются только справочным ориентиром для консультанта при ручном заполнении
`schema/sprint_plan.yaml`.

## Этап 6 — Сборка выходного пакета
**Статус: готово.** Артефакт: `agent/assemble_plan.py` (+ `agent/examples/`).

Исполняемый Python-пайплайн (не только описание правилом, как на Этапах
2/5): Charter → Milestones → WBS → Tasks (из шаблона + вариативность
WBS-6.4 по `docs/selection-rules.md`) → Dependencies (`depends_on` из
шаблона, без изменений) → Оценки трудозатрат (`agent/effort-estimates.yaml`,
Этап 3) → Sprint-план (`agent/sprint-mapping-rules.md`, Этап 5) → Риски
(`agent/risk-register.yaml`, Этап 4, включая R-7 по факту готового
Sprint-плана) → Deliverables (агрегация `verification_checklist`, без
новых формулировок). Sprint-план идёт раньше рисков не случайно: R-7
отбирается по флагу "не укладывается в срок" из Sprint-плана, и без него
этот флаг не существует.

LLM подключается только последним шагом (`adapt_wording_with_llm`) и
только как контракт: разрешено переписывать исключительно текстовые поля
(Charter.description/objective, Task.description,
interview_checklist/verification_checklist); попытка изменить
id/depends_on/used_by/состав Milestone/WBS/Task отклоняется явной
ошибкой (структурный fingerprint до/после сверяется программно), а не
проверяется на словах. Без подключённого LLM-клиента шаг — no-op
passthrough (в репозитории нет живого LLM API).

`--selftest` прогоняет классификацию всех реальных Task шаблона по
`effort-estimates.yaml`, сверяет её с `examples` из самого справочника,
проверяет обе ветки WBS-6.4 (пустой/непустой список интеграций, включая
интеграцию вне `KNOWN_INTEGRATION_DOCS`), ветку "не укладывается в срок"
(R-7) и отказ LLM-контракта при попытке сломать структуру. Пример
прогона на `agent/examples/client-abc.input.json` (тот же условный
клиент "ABC", что фигурировал как пример в `agent/input-schema.json`)
сохранён в `agent/examples/client-abc.plan.json`.

Известное ограничение, найденное при реализации (не исправлялось в этом
проходе — см. `open-issues.md`): у 6 Task keyword-эвристика
`effort-estimates.yaml` не даёт совпадения (глагол не входит в keywords
или не в начале названия) — в `assemble_plan.py` заведён
`MANUAL_TASK_TYPE_OVERRIDES` с обоснованием для каждой, `effort-estimates.yaml`
не менялся.

Этап 8 (Jira-экспорт) в этот пайплайн не входит.

## Этап 7 — Валидатор перед выдачей
**Статус: готово.** Артефакт: `agent/validate_plan.py`.

Линтер над выходом `assemble_plan.py` (Этап 6) — отчёт для консультанта/PM
перед экспортом в Jira (Этап 8), не гейт: ничего не блокирует и не
переписывает, только явно перечисляет находки. Пять проверок: (1)
`integrity` — все ID из `depends_on`/`used_by` существуют среди реальных
Task/WBS ID плана; (2) `source_url` — каждый `source.url` принадлежит
домену/разделу официальной документации Comarch (по домену + форме пути,
т.к. реальные статьи — конкретные страницы, а не сами 4 URL из
`docs/principles.md`); (3) `source` — ни одна Task не без `source`, кроме
`Internal Project Methodology`; (4) `checklist_shape` — `interview_checklist`/
`verification_checklist` строго `list[str]` (автоматизирует находку,
исправленную вручную перед Этапом 7 — незакавыченный `": "` внутри пункта
чек-листа YAML превращает его в dict); (5) `cross_section` — каждая Task
из `milestones` присутствует в `dependencies`, `effort_estimates` и
`sprint_plan.task_sprint`, и наоборот.

`--selftest` намеренно ломает копию `agent/examples/client-abc.plan.json`
под каждую из 5 проверок и подтверждает, что она срабатывает, плюс
regression-прогон на самом `client-abc.plan.json` — ожидается 0 находок.
При реализации найден и исправлен (в `assemble_plan.py`, не в шаблоне)
второй случай той же проблемы, что у `T-6.5.1.depends_on` (Этап 6) —
`T-6.4.1.used_by` тоже ссылался на слот `T-6.4.2` (см. `open-issues.md`
п.4). Отчёт по примеру сохранён в
`agent/examples/client-abc.validation-report.txt`.

Изначально (на момент готовности Этапа 7) регрессия давала 16 находок —
`open-issues.md` п.1 (`used_by` на Milestone ID вместо Task/WBS ID).
Пункт закрыт отдельным проходом: все 16 случаев в `tasks/M*_tasks.yaml`
заменены на конкретного потребителя (Task ID, определённый по
`depends_on` следующей задачи или по текстовой ссылке в `description`,
где явного `depends_on` не было — см. `CHANGELOG.md`), `--selftest`
обновлён под чистую регрессию (0 находок).

## Этап 8 — Jira-экспорт
**Статус: готово.** Артефакт: `agent/jira_export.py` (+ `agent/examples/`).

Экспорт в уже существующий Jira-проект (проект не создаётся этим
скриптом). На вход берётся план Этапа 6 (`assemble_plan.py`),
провалидированный Этапом 7 (`validate_plan.py`) — промежуточных Этапов
7.5/7.6 (документ для согласования с заказчиком и приём правок) в
пайплайне больше нет, они удалены (см. `CHANGELOG.md`): план идёт в
Jira-экспорт как есть, консультант ничего не скачивает и не правит
руками между Этапами 6/7 и 8. Весь план становится одним Epic в этом
проекте, не набором Epic по Milestone:

| Наш уровень | Jira |
|---|---|
| Весь план | 1 Epic (`summary` = `charter.project_name`, `description` = цель/даты/заказчик + Риски + Критерии завершения) |
| Milestone (M1–M9) | Label на каждой Issue/Subtask (`M3`, `M6` и т.п.) |
| WBS | Issue, child Epic — `summary` = название WBS |
| Task (наш) | Subtask под соответствующим WBS-Issue — `summary` = название Task, `description` = Interview/Verification checklist, спринт = имя из `sprint_plan.task_sprint`/`sprint_plan.sprints[].name` (Этап 5/6) |
| `depends_on`/`used_by` | Issue Link `Blocks` (`outward` "blocks" / `inward` "is blocked by"); `used_by` создаёт связь только там, где она не дублирует уже покрытую `depends_on`-парой — `used_by` вторичен по построению (см. `sprint-mapping-rules.md`) и иногда указывает на WBS, а не Task |
| Риски (сработавшие) | Настоящая ADF-таблица (Риск/Затрагивает) в `description` Epic, не отдельные issue |
| Критерии завершения этапов | Настоящая ADF-таблица (ID/Название/Описание: Milestone жирным, затем его WBS) в `description` Epic |
| Гиперподдержка | Отдельный 2-й Epic (`SUPPORT_EPIC_SUMMARY`, по умолчанию "Поддержка") — создаётся **всегда**, без флага, вне зависимости от плана (WBS-9.3 удалён из шаблона, Этап 5). Внутри — ровно одна Task (журнал поддержки), без Subtask/Sprint/timetracking; не участвует в sprint-планировании/R-7/effort-оценках/`--create-sprints` |

Обязательное предусловие перед построением маппинга: скрипт запрашивает у
целевого проекта реальные issuetypes (`GET /rest/api/3/project/{key}`) и
подтверждает, что Subtask доступен на уровне схемы проекта. Если Subtask
недоступен (team-managed проект без Subtask) — скрипт не выбирает
обходной путь молча, а останавливается явной ошибкой с объяснением;
плоская структура (WBS и Task — оба Issue, связаны Issue Link, по
умолчанию `Relates`) включается только по явному `--allow-flat-fallback`.

Явное подтверждение перед созданием issues — отдельный шаг, не
происходящий автоматически вместе со сборкой пакета: по умолчанию скрипт
работает в dry-run (только `GET`-запросы схемы проекта + печать того, что
было бы создано, без единого `POST`); реальное создание — только по
`--execute` вместе с `--confirm` (или интерактивным подтверждением с
итоговыми числами в терминале).

Известное ограничение (v1): нативное поле Jira Sprint принимает не текст,
а ID уже существующего Sprint на Scrum-доске — создание/сопоставление
реальных Sprint через Agile REST API за рамками этого скрипта. Имя
спринта поэтому пишется первой строкой `description` Issue/Subtask, а не
в найденный `customfield_...` — это показывается в dry-run отчёте как
известное решение, а не скрывается.

`--selftest` полностью офлайн (`FakeJiraClient` + фикстуры схемы
проекта, без сетевых вызовов): маппинг WBS→Issue/Task→Subtask на
`client-abc.plan.json`, labels = Milestone ID, Sprint = имя, остановка без
Subtask и явный флаг `--allow-flat-fallback` для плоского режима,
dry-run/`--execute` дают одинаковые счётчики, `skipped_links` пуст на
чистом плане (регрессия), Epic "Поддержка" создаётся всегда (без флага) с
одной Task без Subtask/Sprint/timetracking, и гейт подтверждения перед
`--execute` (`--confirm` / интерактивный ввод / отказ). Начиная с Этапа 5
(шаблон `schema/sprint_plan.yaml`) каждый WBS лежит ровно в одном спринте
по построению — привязка Sprint (`--create-sprints`) не исключает ни
одного WBS как неоднозначного.

Пример прогона: `agent/examples/jira-project-schema.example.json` —
офлайн-фикстура схемы company-managed проекта (заменяет живой Jira-проект,
которого нет в этом репозитории), `agent/examples/client-abc.jira-export.dryrun.log.txt`
— dry-run на `client-abc.plan.json`.

## Этап 9 — Единая точка входа (input → Jira)
**Статус: готово.** Артефакт: `agent/create_project.py`.

Оркестратор поверх уже существующих шагов, а не новая логика: принимает
клиентский `input-schema.json` и параметры подключения к Jira одной
командой и последовательно вызывает `assemble_plan` (Этап 6) →
`validate_plan` (Этап 7, предупреждение, не гейт) → `jira_export` (Этап 8,
`--execute --create-sprints` — опционально, по умолчанию dry-run, как и
раньше в `jira_export.py`). Консультант не скачивает и не передаёт
промежуточный JSON-файл вручную между шагами.

Собранный план всё равно сохраняется на диск — но как внутренний артефакт
аудита (`agent/runs/{client_id}/plan.json`, `agent/runs/{client_id}/jira-
export-state.json` при `--execute`), не как результат для скачивания
консультантом: именно такой артефакт уже спасал при диагностике дублей на
живом TPT (Этап 8, устойчивость к сети) — отказываться от него не нужно,
даже когда `jira_export.py` вызывается не напрямую, а через этот
оркестратор. `client_id` — слаг из `charter.client`, либо явный
`--client-id`.

`--selftest` полностью офлайн (`--schema-fixture` вместо живого
Jira-проекта, во временную директорию): сборка + сохранение `plan.json` по
ожидаемому пути, `validate_plan` — 0 находок (регрессия), dry-run не
создаёт `jira-export-state.json` (только `--execute` пишет его),
`slugify_client_id` на характерном примере.

### Как запустить форму (`agent/run_form.py`)

Разовая настройка: скопировать `agent/.env.example` в `agent/.env` и
заполнить `JIRA_BASE_URL`/`JIRA_EMAIL`/`JIRA_API_TOKEN` (файл в git не
попадает). Дальше на каждый новый проект:

```
python3 agent/run_form.py
```

Открывается браузер на `agent/client-input-form.html` — там же появилось
поле **Jira Project Key** рядом с остальными charter-полями (email/токен в
форме не запрашиваются — берутся сервером из `.env`). Кнопка **«Создать
план (dry-run)»** отправляет заполненные данные локальному серверу
(`POST /plan`), который вызывает `create_project.py` без `--execute` и
показывает текстовый отчёт прямо в форме. После просмотра — кнопка
**«Выполнить (создать в Jira)»** (`POST /execute`, `create_project.py
--execute --confirm`); клик по ней и есть подтверждение человеком. После
ответа на этот запрос лаунчер сам завершает процесс. «Скачать .json» —
прежний fallback, если локальный сервер не поднялся.

---
Изменения в этом плане (новые этапы, переформулировка существующих) фиксируются
здесь же и отражаются короткой записью в `CHANGELOG.md`, как и любое другое
изменение шаблона.
