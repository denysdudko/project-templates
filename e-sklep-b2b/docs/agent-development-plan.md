# План разработки агента-планировщика (Comarch e-Sklep B2B)

Документ фиксирует последовательность этапов создания агента, который берёт
`agent/input-schema.json` конкретного клиента и шаблон Milestones/WBS/Task
(`schema/milestones_wbs.yaml` + `tasks/M*_tasks.yaml`) и генерирует из них
план внедрения. Ничего из перечисленного ниже не меняет утверждённую структуру
шаблона — `schema/milestones_wbs.yaml` и состав `tasks/M*_tasks.yaml` остаются
источником истины для WBS/Task; агент только читает и адаптирует их под клиента
(см. `docs/principles.md`, иерархия источников истины).

Статусы: **готово** / **не начато**. Этап переводится в «готово», когда для него
создан и согласован соответствующий артефакт в `e-sklep-b2b/`.

## Этап 1 — Фиксация входной схемы агента
**Статус: готово.** Артефакт: `agent/input-schema.json`.

Формализует вход клиента (название проекта, заказчик, ERP и его версия,
количество пользователей, интеграции, особенности, даты) как строгий
JSON-контракт — это данные для агента, а не текст, который LLM интерпретирует
свободно.

## Этап 2 — Логика отбора шаблона (не генерация с нуля)
**Статус: готово.** Артефакт: `docs/selection-rules.md`.

Правило маппинга «входные параметры → состав WBS/Task»: core-объём (M1–M9,
WBS-1.1–9.4) неизменен и присутствует всегда; единственная точка вариативности —
WBS-6.4, управляемая полем `integrations` (пустой список — задача-заглушка
T-6.4.2 остаётся framework-only; непустой — на каждую интеграцию разворачивается
child-экземпляр слота T-6.4.2, а не новая WBS). Остальные поля входа (`features`,
`users_count`, `dates`, `erp.version`) — метаданные для контента, не селекторы
структуры.

## Этап 3 — Модель оценки трудозатрат
**Статус: готово.** Артефакт: `agent/effort-estimates.yaml`.

Справочник «тип задачи → базовая оценка часов», отдельный от самого шаблона
задач (в `tasks/M*_tasks.yaml` оценок часов нет и не появляется). Агент
классифицирует конкретную Task по эвристике (ключевые слова в названии +
наличие/характер чек-листов) и подбирает оценку по типу, а не придумывает её
заново для каждой задачи. Источник оценок — опыт консультантов
(`source.type: Internal Project Methodology`), так как документация Comarch
нормативов трудозатрat не содержит.

## Этап 4 — Реестр рисков
**Статус: готово.** Артефакт: `agent/risk-register.yaml`.

Статичная таблица «риск → условие появления по параметрам входа»: 7 рисков,
выведенных из фактических ограничений и зависимостей в M1–M9 (не придуманных
абстрактно), каждый с проверяемым `condition` на полях `agent/input-schema.json`
(`erp.version`, `integrations`, `features`, `users_count`, `start_date`/
`target_launch_date`) и `related_wbs`. Риск неполноты фотоконтента/каталога не
включён — для него нет проверяемого поля во входной схеме (см.
`v2_proposals` в файле реестра). Вероятность/критичность риска не оценивается —
только факт условия появления.

## Этап 5 — Правило Milestone → Sprint
**Статус: готово.** Артефакт: `agent/sprint-mapping-rules.md`.

Единый граф зависимостей по `depends_on` через все M1–M9 (включая
межмилестоновые связи), топологический порядок с тай-брейком по порядку в
шаблоне, жадное распределение по спринтам с трудоёмкостью из
`agent/effort-estimates.yaml` (классификация Task по типу + средняя точка
диапазона часов; отдельные правила для `remediation` — резерв, и
`support_monitoring` — недельная нагрузка гиперподдержки после
`target_launch_date`, не участвующая в проверке укладывания в срок).
Длительность спринта (2 недели) и capacity (1 консультант × 8ч × 5 дней)
зафиксированы как константы v1 — соответствующих полей в
`input-schema.json` нет, добавление не выполнялось без согласования. Если
расчётный объём не укладывается в `target_launch_date` — не блокирует
генерацию, формирует предупреждение с фактическими числами. Номера
спринтов в шаблон не переносятся.

## Этап 6 — Сборка выходного пакета
**Статус: не начато.**

Порядок генерации итогового плана: Charter → Milestones → WBS → Tasks (из
шаблона + отбор Этапа 2) → Dependencies (`depends_on`/`used_by` из шаблона) →
Оценки трудозатрат (Этап 3) → Sprint-план (Этап 5) → Риски (Этап 4) →
Deliverables/критерии завершения этапов (уже есть как `verification_checklist`
на уровне Task). Sprint-план идёт раньше рисков не случайно: риск R-7
(`agent/risk-register.yaml`) отбирается по флагу "не укладывается в срок" из
`agent/sprint-mapping-rules.md`, и без готового Sprint-плана этот флаг ещё не
существует. LLM отвечает только за адаптацию формулировок под клиента
(см. `docs/principles.md`, п. 3 иерархии источников истины), не за структуру
плана.

## Этап 7 — Валидатор перед выдачей
**Статус: не начато.**

Линтер, который проверяет сгенерированный план перед показом человеку:
все ID из `depends_on`/`used_by` существуют в плане; каждый `source.url`
принадлежит одному из разрешённых доменов/разделов документации Comarch
(pomoc.comarchesklep.pl, pomoc.comarch.pl/optima и согласованные в
`docs/principles.md` разделы); ни одна задача не создана без ссылки на
документацию, кроме помеченных `Internal Project Methodology`.

## Этап 8 — Jira-экспорт
**Статус: не начато.**

Маппинг: Milestone → Epic, Task ID → Issue, WBS → Label/Component,
`interview_checklist`/`verification_checklist` → описание задачи в Jira.
Отдельный шаг, срабатывающий только по явному подтверждению утверждённого
плана человеком — не выполняется автоматически вместе со сборкой пакета
(Этап 6).

---
Изменения в этом плане (новые этапы, переформулировка существующих) фиксируются
здесь же и отражаются короткой записью в `CHANGELOG.md`, как и любое другое
изменение шаблона.
