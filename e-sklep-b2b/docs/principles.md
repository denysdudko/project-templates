# Принципы шаблона проекта Comarch e-Sklep B2B

## Иерархия источников истины

1. **Официальная документация Comarch** — основной источник содержания задач, чек-листов и формулировок.
   - https://pomoc.comarchesklep.pl/kategoria/jak-zaczac/
   - https://pomoc.comarchesklep.pl/kategoria/b2b/
   - https://pomoc.comarch.pl/optima/pl/2026_5/dokumentacja/comarch-e-sklep/
   - https://pomoc.comarch.pl/optima/pl/2026_5/dokumentacja/wspolpraca-z-comarch-e-sklep/
2. **Шаблон проекта** — структура работ (WBS) и последовательность (Milestones, зависимости).
3. **Наш опыт / LLM** — только адаптация под конкретного клиента (формулировки, приоритеты, уточнения). Не заменяет и не придумывает структуру.

Структура (`schema/milestones_wbs.yaml`) не меняется при обычном наполнении. Любое предложение, влияющее на архитектуру — Milestones, WBS, состав файлов — фиксируется как предложение для v2, а не вносится текущей правкой.

## Правило WBS / Task

- **WBS = результат.** Отвечает на вопрос "какой результат должен быть получен?".
- **Task = работа.** Отвечает на вопрос "что сделать, чтобы получить этот результат?".
- Task никогда не повторяет название WBS.
- Все Task внутри одной WBS относятся строго к её предметной области; посторонние темы не подмешиваются.
- Каждая Task — законченная работа, которую можно поручить консультанту (провести интервью, проверить конфигурацию, заполнить чек-лист, подготовить отчёт), а не название функции системы.
- Interview Checklist — это содержимое Task, а не отдельная сущность (Task → Interview Checklist).
- Verification Checklist подтверждает результат конкретной Task, а не работу Comarch или системы в целом.
- Каждая Task имеет потребителя: её результат используется следующей работой (`used_by` → `depends_on`).

## Правила чек-листов

- **Interview Checklist** — вопросы заказчику при сборе требований. Каждый пункт самостоятелен и не требует пояснений.
- **Verification Checklist** — проверки после выполнения задачи. Каждый пункт — самостоятельная проверка результата.

## Ограничения процесса

- Не "раздувать" шаблон дополнительными сущностями сверх утверждённой структуры.
- Не стремиться к идеальному покрытию — закрывать содержательные пробелы, не добавлять избыточные слои.
- Изменения вносятся точечно, по одной ограниченной задаче за проход/чат.
