# Логика отбора шаблона (Этап 2)

Документ описывает, как входные параметры проекта (`input_schema.json`)
влияют на состав шаблона `e-sklep-b2b_milestones&wbs.yaml` при генерации
конкретного проекта. Это правила для агента, не изменения структуры
самого шаблона.

## Источник истины

1. Официальная документация Comarch — основной источник:
   - https://pomoc.comarchesklep.pl/kategoria/jak-zaczac/
   - https://pomoc.comarchesklep.pl/kategoria/b2b/
   - https://pomoc.comarch.pl/optima/pl/2026_5/dokumentacja/comarch-e-sklep/
   - https://pomoc.comarch.pl/optima/pl/2026_5/dokumentacja/wspolpraca-z-comarch-e-sklep/
2. Шаблон проекта — структура работ и последовательность (WBS/Task).
3. LLM — адаптация под клиента без изменения базовой логики.

## Базовый принцип

Структура `milestones&wbs.yaml` (M1–M9, WBS-1.1–9.4) — обязательный
core-объём. Она не зависит от входных параметров и присутствует в
любом проекте e-Sklep B2B без исключений и без добавления новых
WBS/Milestone.

Единственная точка вариативности в текущей архитектуре — **WBS-6.4
(Настройка интеграций)**, управляемая полем `integrations` из
`input_schema.json`.

Все остальные поля входного контракта не меняют состав WBS/Task:

| Поле входа       | Влияние |
|-------------------|---------|
| `integrations`    | Управляет составом Task внутри WBS-6.4 (см. ниже) |
| `features`        | Не структурный селектор. Tier-3: LLM адаптирует формулировки описаний / interview_checklist существующих Task под контекст клиента, не выходя за область своего WBS (Правило 3) |
| `users_count`      | Метаданные для контента задач (например, оценки трудозатрат), не влияет на состав WBS/Task |
| `dates`            | Метаданные сроков проекта / задач при выгрузке в Jira |
| `erp.version`      | Метаданные для уточнения версии ERP в описаниях задач |

## Правило для WBS-6.4

Базовые задачи шаблона (файл не меняем):
- **T-6.4.1** — «Уточнить перечень интеграций, подлежащих настройке» — присутствует всегда.
- **T-6.4.2** — «Настроить согласованные интеграции согласно их официальной документации» — обобщённая задача-заглушка (framework-only).

### Если `integrations` пуст

WBS-6.4 остаётся как в шаблоне: только T-6.4.1 и T-6.4.2, без
добавлений.

### Если `integrations` не пуст

При генерации проекта (не в файле шаблона) T-6.4.1 остаётся без
изменений. Вместо одной обобщённой T-6.4.2 агент разворачивает по
одной конкретной Task на каждую интеграцию из списка — это
child-экземпляры слота T-6.4.2, а не изменение структуры шаблона.

Формат каждой такой задачи:

- **id**: `T-6.4.2.1`, `T-6.4.2.2`, ... — по порядку элементов в
  `integrations`.
- **name**: «Настроить интеграцию `<Название>` согласно документации
  Comarch» (Task не повторяет WBS — Правило 2; конкретная законченная
  работа — Правило 4).
- **depends_on**: `[T-6.4.1]`. Ветки по разным интеграциям независимы
  друг от друга.
- **source.type**: `Official Comarch Documentation`.
- **source.url**: конкретная статья документации именно про эту
  интеграцию (не общая `/kategoria/integracje/`) — находится и
  сверяется по документации Comarch, не придумывается.
- **interview_checklist**: заполняется, только если для настройки
  этой интеграции по документации требуются данные от заказчика
  (ключи API, идентификаторы аккаунта и т.п.); иначе `[]`.
- **verification_checklist**: минимум один пункт «Проверить, что
  интеграция `<Название>` активна и работает согласно её
  документации», плюс проверки, специфичные для неё, если они
  описаны в документации.
- **used_by**: `[WBS-6.5]` — как у базовой T-6.4.2.

Никаких новых ключей схемы не вводится — используются только
существующие поля Task (id, name, description, performer,
depends_on, source, result, used_by, interview_checklist,
verification_checklist). Название интеграции указывается только в
`name`/`description`.

## Границы

- Это правило не меняет `e-sklep-b2b_milestones&wbs.yaml`.
- Любое потенциальное расширение (например, аналогичная
  вариативность в других WBS) не вводится «по ходу» — выносится как
  предложение для v2, если затрагивает утверждённую архитектуру.
