## 3. `T-6.5.1.depends_on` и `T-6.4.1.used_by` ссылаются на слот `T-6.4.2`, который при непустых `integrations` не существует
**Статус: ЗАКРЫТО.** Обработано на верном архитектурном уровне (генерация плана,
не шаблон), покрыто регрессией в обоих `--selftest` (`assemble_plan.py` и
`validate_plan.py`) и подтверждено: `validate_plan.py` (`integrity`) — 0 находок
на пересобранном `agent/examples/client-abc.plan.json`. Дальнейших действий не
требуется, если у WBS-6.4 не появится ещё один аналогичный "слот" (см. заметку
в коде `patch_stale_slot_dependencies`).

Когда WBS-6.4 разворачивается в `T-6.4.2.1..N` (`docs/selection-rules.md`),
единственные в шаблоне Task, ссылающиеся на `T-6.4.2` (`T-6.5.1.depends_on` и
симметрично `T-6.4.1.used_by`), теряют эту ссылку, если её не переписать на
сгенерированные id — иначе граф зависимостей просто не замечает пропавший id,
а не сигналит об ошибке. Второй случай (`T-6.4.1.used_by`) найден Этапом 7
(`validate_plan.py`, проверка `integrity`), не сразу при реализации Этапа 6.

Не исправлено в `tasks/M6_tasks.yaml` (файл не менялся) — обработано в
`agent/assemble_plan.py` функцией `patch_stale_slot_dependencies` (для обоих
полей, `depends_on` и `used_by`) во время генерации плана, с регрессионной
проверкой в `--selftest` (и в `--selftest` у `validate_plan.py`). Если у
WBS-6.4 в будущем появится ещё один аналогичный "слот", эту функцию нужно
будет обобщить, а не копировать точечно.

## 4. Jira-экспорт: duedate/startDate Issue/Subtask по трудоёмкости не проставляются
**Статус: ОТКРЫТО (backlog).** Не начиналось — зафиксировано как предложение,
не реализовывалось в этом проходе.

`agent/jira_export.py` (Этап 8) при `--create-sprints` привязывает WBS-Issue к
реальному нативному полю Sprint (`POST /rest/agile/1.0/sprint/{id}/issue`), но
не проставляет `duedate`/датy начала на самих Issue/Subtask — единственный
источник дат для консультанта сейчас: имя/даты спринта (`sprint_plan.sprints[].
name`/`start_date`/`end_date`, Этап 5 v2) и текст "Sprint: ..." в description.
Проставление `duedate` (и стартовой даты, если для неё найдётся подходящее
поле Jira) по фактической трудоёмкости Task внутри спринта — отдельная,
более объёмная задача (нужно решить, как распределять даты внутри спринта
между несколькими Task одного WBS/Subtask), не вводится этой правкой.

## 5. `--lang` (UA-вариант контента, CHANGELOG v3.6): архитектурная правка внесена напрямую, не как v2-предложение
**Статус: ОТКРЫТО (осознанное решение, не забытое).**

`docs/principles.md`/`CLAUDE.md` требуют фиксировать изменения состава файлов
шаблона и новых ключей схемы как предложение для v2, а не вносить текущей
правкой. `--lang`/`schema/milestones_wbs.ua.yaml`+`tasks/M*_tasks.ua.yaml`/
поле `lang` в `agent/input-schema.json` — ровно такое изменение (новый файл
шаблона на язык + новый ключ схемы), но было явно запрошено и подтверждено
пользователем с полным пониманием конфликта с процессом (см. историю сессии).
Зафиксировано здесь, а не тихо: следующий аналогичный язык (например,
`schema/milestones_wbs.en.yaml`) должен подниматься как отдельное решение, не
автоматически по прецеденту этой правки.

Дополнительно найдено и исправлено по ходу (не отдельный open issue, т.к.
закрыто в этой же правке): keyword-эвристика `effort-estimates.yaml`
(`classify_task`) сопоставляет русские глаголы с текстом `Task.name` —
без исправления `build_effort_estimates` падал бы на любой Task при
`--lang ua`. Классификация теперь всегда идёт по RU-названию Task (по ID),
не по отображаемому тексту — см. CHANGELOG v3.6.

## 6. Jira-экспорт (`agent/jira_export.py`): заголовки/подписи ADF-отчёта не локализованы для `--lang ua`
**Статус: ОТКРЫТО (backlog).**

При `--lang ua` содержимое Epic/Issue/Subtask (названия Milestone/WBS/Task,
чек-листы) идёт на украинском — оно приходит из уже собранного `plan`
(Этап 6). Epic "Поддержка"/"Підтримка" локализован отдельно
(`SUPPORT_EPIC_SUMMARY_BY_LANG` и т.п., `plan["meta"]["lang"]`). Но
"обвязка" отчёта — заголовки ADF-таблиц ("Риск"/"Затрагивает"/"ID"/
"Название"/"Описание"), подписи в description Эпика ("Заказчик:", "Цель:",
"Дата начала:", "Риски проекта" и т.п.) — хардкод на RU в коде
`jira_export.py`, не читается ни из `plan`, ни из lang-словаря. При
`--lang ua` получится смешанный отчёт: контент на украинском, обвязка
таблиц/подписей — на русском. Не исправлено в этой правке — не входило в
исходный запрос (только Milestones/WBS/Task/Support Epic), объём сопоставим
с отдельной задачей (перевести и параметризовать по lang все статические
строки Jira-экспорта, не только контентные).
