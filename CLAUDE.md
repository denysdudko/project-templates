# CLAUDE.md

Руководство для Claude Code при работе в этом репозитории.

## Что это за репозиторий

`project-templates` — коллекция переиспользуемых шаблонов проектов (Milestones/WBS/Task),
чек-листов и документации для AI-driven внедрений ПО и ERP-систем. Каждый шаблон
лежит в своей директории верхнего уровня (сейчас: `e-sklep-b2b/`) и описывает
типовой проект целиком — от анализа требований до запуска и гиперподдержки —
независимо от конкретного клиента.

Шаблон — это не документация одного проекта, а порождающая структура: агент
берёт `input_schema.json` конкретного клиента и шаблон Milestones/WBS/Task и
генерирует из них конкретный план внедрения.

## Структура шаблона (на примере `e-sklep-b2b/`)

```
e-sklep-b2b/
  schema/milestones_wbs.yaml   # структура: Milestones (M1–M9) → WBS. Меняется редко.
  tasks/M{1..9}_tasks.yaml     # содержимое: Task с чек-листами внутри каждого WBS
  agent/input-schema.json      # входной контракт агента-планировщика (параметры клиента)
  docs/principles.md           # правила WBS/Task/чек-листов, иерархия источников истины
  docs/selection-rules.md      # как input_schema влияет на состав шаблона при генерации
  docs/source-map.md           # таблица WBS → фактические ссылки на документацию Comarch
  CHANGELOG.md                 # история изменений шаблона по версиям
  open-issues.md                # известные несоответствия, не блокирующие работу
```

Каждый следующий шаблон в этом репозитории (для другого продукта/ERP) должен
следовать этой же файловой структуре.

## Иерархия источников истины

Обязательна для любого шаблона в этом репозитории:

1. **Официальная документация вендора** (для e-Sklep — Comarch: pomoc.comarchesklep.pl,
   pomoc.comarch.pl) — основной источник содержания задач, чек-листов и формулировок.
2. **Шаблон проекта** — структура работ (WBS) и последовательность (Milestones, зависимости).
3. **Опыт / LLM** — только адаптация под конкретного клиента (формулировки, приоритеты,
   уточнения). Никогда не заменяет и не придумывает структуру.

Там, где вендор не описывает процесс (интервью, анализ требований, приёмка,
управление проектом), задача помечается `source.type: Internal Project Methodology`,
а не приписывается официальной документации без URL.

## Правило WBS / Task

- **WBS = результат.** Отвечает на вопрос «какой результат должен быть получен?».
- **Task = работа.** Отвечает на вопрос «что сделать, чтобы получить этот результат?».
- Task никогда не повторяет название своего WBS.
- Все Task внутри одного WBS относятся строго к его предметной области; посторонние
  темы не подмешиваются.
- Каждая Task — законченная работа, которую можно поручить консультанту (провести
  интервью, проверить конфигурацию, заполнить чек-лист, подготовить отчёт), а не
  название функции системы.
- Interview Checklist — это содержимое Task, а не отдельная сущность (Task →
  Interview Checklist).
- Verification Checklist подтверждает результат конкретной Task, а не работу
  вендора/системы в целом.
- Каждая Task имеет потребителя: её результат используется следующей работой
  (`used_by` должен указывать на Task ID, а не на Milestone/WBS ID — см. `open-issues.md`
  для известных исключений).

## Правила чек-листов

- **Interview Checklist** — вопросы заказчику при сборе требований. Каждый пункт
  самостоятелен и не требует пояснений.
- **Verification Checklist** — проверки после выполнения задачи. Каждый пункт —
  самостоятельная проверка результата.

## Ограничения процесса

- Структура (`schema/milestones_wbs.yaml`) не меняется при обычном наполнении
  контентом. Любое предложение, влияющее на архитектуру — Milestones, WBS, состав
  файлов, новые ключи схемы — фиксируется как предложение для следующей версии
  (v2), а не вносится текущей правкой.
- Не «раздувать» шаблон дополнительными сущностями сверх утверждённой структуры.
- Не стремиться к идеальному покрытию — закрывать содержательные пробелы, не
  добавлять избыточные слои.
- Изменения вносятся точечно, по одной ограниченной задаче за проход/чат.
- Правила вариативности шаблона (что зависит от входных параметров клиента, а что
  нет) описываются в `docs/selection-rules.md` конкретного шаблона и не меняют
  сам файл структуры.

## Работа с этим репозиторием

- Перед изменением содержимого Task/WBS в существующем шаблоне — прочитать
  `docs/principles.md` и `docs/selection-rules.md` этого шаблона.
- После изменения ссылок на документацию — обновить `docs/source-map.md`.
- Любое изменение шаблона — короткая запись в `CHANGELOG.md` (что изменено и почему).
- Известные несоответствия, которые не блокируют текущий этап, фиксируются в
  `open-issues.md`, а не исправляются попутно вместе с несвязанной задачей.
